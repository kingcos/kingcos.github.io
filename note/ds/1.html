视频 4

笔记：

数据结构：
数据对象在计算机中的组织方式（例如图书的摆放方法）

  • 逻辑结构（eg: 线性表，树，图）
  • 物理存储结构（内存中具体的存储方式，eg: 数组，链表）

数据对象必定与一系列加在其上的操作相关联（例如查找或增加图书）

  • 完成这些操作所用的方法就是算法
  • 描述数据结构：
    抽象数据类型（Abstract Data Type）

数据类型：

  1. 数据对象集
  2. 数据集合相关联的操作集

抽象：描述数据类型的方法不依赖于具体实现

  • 与存放数据的机器无关
  • 与数据存储的物理结构无关
  • 与实现操作的算法和编程语言均无关

即只描述数据对象集”是什么”，并不涉及”如何做到”的问题。

eg:

类型名称：矩阵（Matrix）
数据对象集：
一个 M * N 的矩阵由 M * N 个三元组 (a, i, j) 构成，其中 a 是矩阵元素的值，i 是元素所在的行号，j 是元素所在的列号
操作集：
对于任意矩阵 A, B, C 属于 Matrix，以及整数 i, j, M, N

  1. Matrix create(int M, int N); 返回一个 M * N 的空矩阵
  2. int GetMaxRow(Matrix A) 返回矩阵 A 的总行数
  3. int GetMaxCol(Matrix A) 返回矩阵 A 的总列数
  4. ElementType getEntry(Matrix A, int i, int j); 返回矩阵第 i 行，第 j 列的元素；
  5. Matrix Add(Matrix A, Matrix B) 如果 A 和 B 的行、列数一致，则返回矩阵 C = A + B，否则返回错误标志
  6. Matrix Multiply(Matrix A, Matrix B) 如果 A 的列数等于 B 的行数，则返回矩阵 C = A * B，否则返回错误标志

讨论题：

问：抽象有什么好处？
任何事物存在都要有个理由，为什么大家这么稀饭”抽象”？

答：

抽象，意味着不具体。

  1. 抽象使得不涉及具体的程序设计语言。我认为，语言只是工具，而思想是互通的，即一种思想可以为各种语言所用，只是代码略有不同，所实现的平台等有区别
  2. 抽象使得我们能够更加独立地思考，而不是禁锢在某一门语言里。这样，把思想独立的提取出来，当我们需要具体实现的时候，就可以随机应变
  3. 当然，这样也方便了描述，适应性也更强

书摘：

理解”抽象”的含义：

  1. 忽略元素类型等细节问题，使得适用于任何一种类型；
  2. 对数据对象的描述不依赖于其在计算机中具体的存储方法；
  3. 在描述操作时，只描述这个操作是做什么的，并不涉及操作的具体实现方法，更与实现操作的编程语言没有关系。
    综上，抽象数据类型描述的重要特征是”抽象”。抽象是计算机求解问题的基本方式和重要手段，它使得一种设计可以应用于多种场景。而且通过抽象可以屏蔽底层的细节，使设计更加简单、理解更加方便。
    抽象数据类型的描述方法和面向对象的思想是一致的，它把数据对象和相关操作封装在一起，对于需要调用这个数据类型的用户而言，无论内部的具体实现如何改变，只要对外描述的接口不变，就不影响使用。

什么是算法

视频 1

笔记：

算法（Algorithm）

  • 一个有限指令集
  • 接受一些输入（有些情况下不需要输入）
  • 产生输出
  • 一定在有限步骤之后终止（不可无限循环）
  • 每一条指令必须
    • 有充分明确的目标，不可以有歧义
    • 计算机能处理的范围之内
    • 描述应不依赖于任何一种计算机语言以及具体的实现手段

抽象：

  • List 到底是数组还是链表（虽然看上去很像数组）？
  • Swap 用函数还是用宏去实现？
    这些就不是描述算法时所关心的

书摘：

算法不是程序：

  1. 程序可以无限运行（例如操作系统），但算法必须在有限步后终止
  2. 算法比程序”抽象”，强调表现”做什么”，而忽略细节性的”怎么做”。这样的好处是使整体思路清晰易懂，形成模块化的风格，这在注重团队配合的软件开发过程中显得特别重要

视频 2

笔记：

空间（Space）复杂度 $S(n)$

  • 根据算法写成的程序在执行时占用存储单元的长度
  • 这个长度往往与输入数据的规模有关
  • 空间复杂度过高的算法可能导致使用的内存超限，造成程序非正常中断

时间（Time）复杂度 $T(n)$

  • 根据算法写成的程序在执行时耗费时间的长度
  • 这个长度往往也与输入数据的规模有关
  • 时间复杂度过高的低效算法可能导致我们在有生之年都等不到运行结果

什么是好的算法？

  • 在分析一般算法的效率时，我们经常关注下面两种复杂度
    • 最坏情况复杂度 $T_{worst}(n)$
    • 平均复杂度 $T_{avg}(n)$
      $T_{avg}(n){\le}T_{worst}(n)$
  • 在分析时，通常分析最坏情况复杂度 $T_{worst}(n)$，因为：平均复杂度平常不是一个容易搞定的问题，所以分析平均复杂度会很难

eg:

for 循环：

  • 空间占用永远是个常量，不会随着 n 的增长而增长对内存的占用（不涉及程序调用）

递归：

  • 当n越大时，$S(n)$ 呈线性增长，而这个函数所占空间是有限的，当把他的有限空间用完，程序就会崩溃

讨论题：

问：分析”二分法”
查找算法中的”二分法”是这样定义的：
给定 N 个从小到大排好序的整数序列 List[]，以及某待查找整数 X，我们的目标是找到 X 在 List 中的下标。即若有 List[i] = X，则返回 i；否则返回 -1，表示没有找到。
二分法是先找到序列的中点 List[M]，与 X 进行比较，若相等则返回中点下标；否则，若 List[M] > X，则在左边的子系列中查找 X；若 List[M] < X，则在右边的子系列中查找 X。
试写出算法的伪码描述（注意要会抽象地表达哦，不要过分依赖于具体的语言和实现细节），并分析最坏、最好情况下的时间、空间复杂度。

答：

最好的情况下，那么就是恰好这个 X 在整数列中间：

$T(n)=O(1)$, $S(n)=O(1)$

最坏的情况下，那么就是最后在查找的最后一次，是这个 X：

$T_{worst}(n)=O({log}\,n)$, $S_{worst}(n)=O({log}\,2n)$

书摘：

上述递归算法非正常中断的关键原因：
计算机在一个函数 A 内部处理另一个函数 B 的调用时，必须先把 A 的当前状态保存在内存中，当 B 的调用完成后，再释放内存恢复状态，继续执行 A 的其余语句
所以递归中，直到执行 PrintN(0) 时，函数才返回，系统开始逐级释放内存，假设存储每个函数状态需占用 1 个单位内存，那么执行 PrintN(N) 需要 N 个单位，当 N 非常大时，计算机内存不足就会造成程序崩溃

视频 3

笔记：

复杂度的渐进表示法（这里只用 $T(n)$ 演示，$S(n)$ 同理）

  • $T(n)=O(f(n))$ 表示存在常数 $C{\gt}0$, $n_0{\gt}0$ 使得当 $n{\gt}n_0$ 时有 $T(n){\le}C{\cdot}f(n)$，即 $O(f(n))$ 表示 $f(n)$ 是 $T(n)$ 的某种上界（对于充分大的 $n$ 而言）
  • $T(n)={\Omega}(g(n))$ 表示存在常数 $C{\gt}0$, $n_0{\gt}0$ 使得当 $n{\ge}n0$ 时有 $T(n){\ge}C{\cdot}g(n)$ 即 ${\Omega}(f(n))$ 表示 $g(n)$ 是 $T(n)$ 的某种下界（对于充分大的 $n$ 而言）
  • $T(n)={\Theta}(h(n))$ 表示同时有 $T(n)=O(h(n))$ 和 $T(n)={\Omega}(h(n))$ 既是上界又是下界

注意：一个函数上界和下界非唯一，而是有无限个，但是我们在写的时候是我们能找的尽可能的最小上界或者最大下界

$$O(1){\lt}O({log}\,n){\lt}O(n){\lt}O(n{log}\,n){\lt}O(n^2){\lt}O(n^3){\lt}O(2^n){\lt}O(n!)$$

复杂度分析小窍门：

  • 若两段算法分别有复杂度 $T_1(n)=O(f_1(n))$ 和 $T_2(n)=O(f_2(n))$，则：
    1. 两段算法串联：$T_1(n)+T_2(n)=max(O(f_1(n)),O(f_2(n))) $，较慢的算法决定了最终的效率
    2. 两端算法嵌套：$T_1(n)*T_2(n)=O(f_1(n)*f_2(n))$，两者的乘积决定了最终的效率
  • 若 $T(n)$ 是关于 $n$ 的 $k$ 阶多项式，那么 $T(n)=\Theta(n^k)$ （考虑其中最大项）
  • 一个 for 循环的时间复杂度等于循环次数乘以循环体代码的复杂度；
  • 若干层嵌套循环的时间复杂度等于各层循环次数的乘积再乘以循环体代码的复杂度
  • if-else 结构的复杂度取决于 if 的条件判断复杂度和两个分枝部分的复杂度，总体复杂度取三者中最大

书摘：

精确地比较程序执行的步数是没有意义的，因为每步执行时间可能不同，例如递归调用的 “1” 步，实际上涉及对系统堆栈的很多处理，比循环的 “1” 步计算慢很多。所以只考虑宏观渐进性质，即当输入规模 $n$ “充分大”时，观察不同算法复杂度的”增长趋势”，以判断哪种算法必定效率更高
$T(n)=O(f(n))$
$n$: 问题规模
$f(n)$: 关于 $n$ 的函数
推导：

  1. 常数阶：$O(1)$ 用常数 $1$ 取代运行时间中的所有加法常数
  2. 在修改后的运行次数函数中，只保留最高阶项
  3. 若最高阶项存在且不为 $1$，则去掉与这个项相乘的常数

分而治之：把一个大的复杂问题切分成小的块，分头去解决，最后再把结果合并起来

$T(N)=O(Nlog\,N)$

复杂度计算过程：

$$\begin{align}
T(N)&=2T(N/2)+cN, T(1)=O(1)\\
&=2[2T(N/{2^2})+cN/2]+cN\\
&=2^2[2T(N/{2^3}+cN/{2^2})]+cN\\
&\cdots\\
&=2^kO(1)+ckN\quad(N/{2^k}=1)\\
&=O(Nlog\,N)
\end{align}$$

书摘：

分治法：

  1. 将序列从中分为左、右两个子序列
  2. 递归求得两子列的最大和 $S_l$ 和 $S_r$
  3. 从中分点分别向左、右两边扫描，找出跨过分界线的最大子列和 $S_m$
  4. $S_{max}=max{\lbrace}S_l,S_m,S_r{\rbrace}$

讨论题：

问：算法 3 的空间复杂度是多少？
这里要考虑 2 个问题：

  1. 由于递归而产生的空间复杂度是多少？
  2. 算法的整体空间复杂度一共是多少？

参考答案：

一：
这个问题是问一共递归了多少层。由于我们每次递归都把搜索范围缩小一半，也就是 $N/2/2/2{\cdots}$ 直到得到 1，于是由公式 $N/2^k=1$，推出 $k={log}\,N$，即递归最多需要 $O({log}\,N)$ 次，占用的空间跟递归的次数成正比，也就是 $O({log}\,N)$

二：
由于 List 是作为指针参数传递的，函数内部并没有声明额外的、长度跟 $N$ 有关的数组，所以每次调用函数都只占用 $O(1)$ 的额外空间，于是占用的空间跟递归的次数成正比，也是 $O({log}\,N)$

如果考虑整个 main 函数的空间复杂度，就要把原始 List 也算在内，它占了 $O(N)$ 的空间，那么整体空间复杂度就应该是 $O(N)+O({log}\,N)=O(N)$

注意：如果每次传递的不是数组的指针，而是每次复制一段数组，那么递归产生的空间复杂度就变成了$O(N)$

小结

书摘：

  1. “数据结构”包括数据对象集以及它们在计算机中的组织方式，即它们的逻辑结构和物理存储结构，同时还包括与数据对象集相关联的操作集以及实现这些操作的最高效算法。抽象数据类型是用来描述数据结构的重要工具
  2. “算法”是解决问题步骤的有限集合，通常用某一种计算机语言进行伪码描述。我们用时间复杂度和空间复杂度来衡量算法的优劣，用渐进表示法分析算法复杂度的增长趋势
