[译]摊平由「try?」造成的嵌套可选

2019-07-08
#Translation  #iOS  #Swift 
  • 提案: SE-0230
  • 作者: BJ Homer
  • 审查管理员: John McCall
  • 状态: 已实现 (Swift 5)
  • 实现: apple/swift#16942
  • 审查: (论坛帖子) (验收)

介绍

Swift 中的 try? 语句目前很容易引入嵌套可选（译者注：嵌套可选即类似 var foo: String?? 可选的可选类型）。而用户难以推断嵌套可选的产生原因，所以 Swift 尝试避免在一些常见情况下产生嵌套可选。

该文档提议给予 try? 与其它常见的 Swift 功能中相同的可选摊平行为（译者注：可选摊平即将可选类型的值去掉可选的一层），来避免常见的嵌套可选。

Swift-evolution 帖子: 使 try? 与可选链摊平协同工作

动机

目前，使用 try? 非常容易产生嵌套的 Optional 类型。虽然构建嵌套可选是被允许的，但这通常并非是开发者所希望的。

Swift 拥有许多机制来避免意外创建嵌套可选。比如：

// 注意 as? 是如何可以无视被转换的值是否是可选类型的，其总是产生了相同的类型
let x = nonOptionalValue() as? MyType    // x 是 'MyType?' 类型
let y = optionalValue() as? MyType       // y 是 'MyType?' 类型

// 注意可选链可以无视调用者是否产生可选值，其总是产生了相同的类型
let a = optThing?.pizza()             // a 是 'Pizza?' 类型
let b = optThing?.optionalPizza()     // b 是 'Pizza?' 类型

但是 try? 的行为不同：

let q = try? harbor.boat()           // q 是 'Boat?' 类型
let r = try? harbor.optionalBoat()   // r 是 'Boat??' 类型

上述例子是特意举例的，但其实在生产环境的代码中，也很容易造成嵌套可选。举个例子：

// 由于 'foo' 的可选链，'foo?.makeBar()' 的结果是 'Bar?'。
// 'try?' 增加了额外一层可选。所以 'x' 类型是 'Bar??'
let x = try? foo?.makeBar()

// JSONSerialization.jsonObject(with:) 返回 'Any'。
// 我们使用 'as?' 来确认结果是否是期望的类型，但 'try?' 增加了额外一层可选，导致结果 'dict' 现在是 '[String: Any]??' 类型。
let dict = try? JSONSerialization.jsonObject(with: data) as? [String: Any]

虽然使用 try? 可以相当简单地创建一个嵌套的可选类型，现存代码的一个调查建议是这并非是理想的结果。使用 try? 和嵌套可选的代码几乎总是伴随着以下几种模式之一：

// 模式 1：双层 if-let 或 guard-let
if  let optionalX = try? self.optionalThing(),
    let x = optionalX {
    // 在这里使用 'x'
}

// 模式 2：引入括号以使用 'as?' 为我们摊平（嵌套可选）
// Pattern 2: Introducing parentheses to let 'as?' flatten for us
if let x = (try? somethingAsAny()) as? JournalEntry {
    // 在这里使用 'x'
}

// 模式 3：模式匹配
if case let x?? = try? optionalThing() {
    // 在这里使用 'x'
}

这些变通方法的需要使得语言更加难以学习和使用，而且真的没有任何好处回报。

使用 try? 的代码通常不关心错误和空结果的区别，这就是为什么上述的模式只聚焦于值而忽略错误。如果开发者真的关系如何探测错误，他们可能会使用 do/try/catch 取而代之。

提议解决方案

在 Swift 5 中，try? someExpr() 将和 foo?.someExpr() 一致：

  • 如果 someExpr() 制造非可选值，其将会被包在可选值内。
  • 如果 someExpr() 制造 Optional，那么将不会附加可选。

这导致以下的 try? 表达式类型改变：

// Swift 4: 'Int??'
// Swift 5: 'Int?'
let result = try? database?.countOfRows(matching: predicate)


// Swift 4: 'String??'
// Swift 5: 'String?'
let myString = try? String(data: someData, encoding: .utf8)

// Swift 4: '[String: Any]??'
// Swift 5: '[String: Any]?'
let dict = try? JSONSerialization.jsonObject(with: data) as? [String: Any]

当子表达式制造非可选类型时，整体类型将不会发生改变。

// Swift 4: 'String?'
// Swift 5: 'String?'
let fileContents = try? String(contentsOf: someURL)

如果子表达式已经制造了嵌套的可选类型，结果将和嵌套的（层级）一致：

func doubleOptionalInt() throws -> Int?? {
    return 3
}

// Swift 4: 'Int???'
// Swift 5: 'Int??'
let x = try? doubleOptionalInt()

关于 try? 和 as? 的额外说明

虽然 as? 经常有摊平可选类型的影响（如上在「动机」部分的举例），但它与这次的 try？ 并非是完全相同的行为。因为 as? 采用的是明确类型，其实际上可以摊平多个层级的嵌套可选。foo as? T 将总是制造 Optional<T> 而无论 foo 上有多少可选。这可能潜在地增加或减少可选的级别，取决于指定的类型。（它同样可以转换子类型与父类型之间，这与所考虑的行为无关。）

在实践中，as? 与嵌套可选最常使用在降低 T?? 到 T?，因此使得其外在看来与使用可选链，以及提出的 try? 类似。但是 as? 是一个比这里提出的 try? 更加有力，且多才多艺的结构。

细节设计

在 Swift 4 中，try? 表达式的类型被定义为 Optional<T>，T 表示 try? 之后的表达式的类型。

在 Swift 5 中，try? 的类型将变为 U，U 是 Optional<_> 类型且子表达式类型 T 可以强制转换为 U。类型约束系统将自动选择满足约束的最小嵌套可选等级，因此导致了本提案中描述的行为。

泛型

关于与泛型互通的一些问题被提出，如下例：

func test<T>(fn: () throws -> T) -> T? {

    // 如果 T 是可选类型，该行是否会改变行为？
    if let result = try? fn() {
        print("We got a result!")
        return result
    }
    else {
        print("There was an error")
        return nil
    }
}

// T 在这里被推断为 'Int'
let value  = test({ return 15 })

// T 在这里被推断为 'Int?'
let value2 = test({ return 15 as Int? })

答案是这与 T 在运行时是否可选无关。在编译时刻，result 拥有明确的类型：T。这在 Swift 4 和 Swift 5 的模式中都是正确的，因为 T 在编译时刻无法得知是否是 Optional 类型，try? 表达式添加了一层 Optional，之后通过 if let 解包。

使用 try? 的泛型代码可以依然继续使用，而无需关心泛型类型可能在运行时变成可选。这种情况的行为并没有发生改变。

源代码兼容

_如果自身没有明确摊平可选_，却操作可选的子表达式，这对于 try? 表达式来说是一种突破性的改变。尽管出现的情况很少，但详见下面的分析。当编译器运行在 Swift 4 模式下，可以提供向后兼容的行为，以及迁移应当适用于大多数常用情况。

关于 Swift 5 中包含突破性变化的讨论很激烈，但我相信它能通过。以下是 swift-evolution README 中列出的标准：

1. 当前的语法或 API 必须主动展示可能为用户带来的问题

嵌套可选是一个复杂的概念。它们在语言中有份量，但它们的使用应当是有目的的。迫于我们目前对于 try? 的交互以及可选链或者 as? 转换，初学者非常容易不明所以地创造在出嵌套可选。

使用 try? 来实际检测错误比理解只使用 try 的代码要*更加*晦涩。对比：

// 使用 'try?'
if let result = try? foo?.bar() {
    // 额外对 'result' 做些什么，尽管在 if let 内其仍然可能为 nil
}
else {
    //出现错误，但我们不清楚原因
    // There was an error, but we don't know what it is
}

// 使用 'try/catch'
do {
    let result = try foo?.bar()
    // 额外对 'result' 做些什么，由于可选链其可能为 nil
}
catch {
    // 处理错误
}

使用 try? 的变体将非常不清晰（result 是什么类型？），且没有显而易见的好处。在不关心 else 语句且只希望处理存在的值时使用 try? 更好，但提议的改变将更好的作用于这种使用情况。

由于与 as? 转换的交互，当前的语法仍比较痛苦，如下：

if let x = try? foo() as? String {
    // 由于我们将 `String` 视作这里想要的类型，
    // 所以意外的是 `x` 是 `Optional<String>` 类型
}

2. 新语法 / API 必须更加清晰且不与现存的 Swift 语法冲突

以上提出的变更更好地解决了所有的问题。此次变更同样明确了 try? 的角色，即，而非 try/catch 类似的错误处理机制。

3. 已存在的代码必须拥有通过合理地自动迁移路径

如下分析，大多数源代码将无需迁移；已经遇到产生嵌套可选的开发者可能已经使用了包含该变动的源代码兼容的模式。该提案简单地提供了简化代码的方式。

自动迁移已为双 if/guard let 和 case let value??: 上述模式实现。

Swift 源兼容性套件分析

Swift 源兼容性套件表明这对于大多数用户来说并非是一项巨变。我手动检查了 try? 在兼容套件中的使用案例。以下是结果：

  • 兼容性套件中总共有 613 个 try? 的实例。其中绝大多数出现在使用非可选子表达式中，并且不受该提议的影响。

  • try? ... as? 的实例有 4 个。所有这四个都将 try? 包裹在括号中以获得摊平行为的 as?，且他们与该改动是源代码兼容的。他们看来这样：

  (try? JSONSerialization.jsonObject(with: $0)) as? NSDictionary

  • 横跨三个项目，共有 12 个 try? foo?.bar() 的情况。 其中 10 个被分配到 _ = try? foo?.bar()，所以结果的类型不重要。 其中 2 个情况拥有一个 Void 子表达式类型，且没有赋值结果到任何变量。

  • 有 6 个 try? somethingReturningOptional() 实例。他们全都使用 flatMap { $0 } 手动摊平，且因此和此变更源代码兼容，因为该表达式的返回类型在任一行为下都是相同的。 They all flatten it manually using flatMap { $0 }, and are thus source-compatible with this change, as the return type of that expression is the same under either behavior.

  (try? get(key)).flatMap { $0 }

  • 据我所知，整个套件中没有任何双可选用来区分错误情况和空值转为值的情况。

  • 据我所知，在兼容套件中没有任何源代码不兼容的情况。

ABI 稳定性的影响

对 ABI 无影响。

API 适应性的影响

try? 表达式永远不会暴露在函数边界，因此 API 适应性应当不受影响。

考虑过的替代方案

改变 try? 的绑定优先权

如 let x = try? getObject() as? Foo 类似的表达式，可以通过转换表达式为 (try? getObject) as? Foo，来消除嵌套可选。添加明确的括号已经成为双可选问题的常见变通方法。

但是，这项改变不会解决 try? 与可选链（例如：try? foo?.bar?.baz()）的情况，以及不会解决从函数直接返回的一个可选结果（例如：try? cachedValue(for: key)）。

改变 try? 的绑定优先权也可能会对源码破坏比该提案更深远。

什么也没做

在当前的模式下写出正确的代码是可以的。我们并非提出完全从语言中移除嵌套可选，我们只是希望用户将其弄清楚。

这是一个可行的解决方案，但作为语法糖的一部分，为简化日常使用却拥有如此复杂的结构是很奇怪的。
